## Introduction {/* #introduction */}

Welcome to nanos world! A sandbox multiplayer game that allows you to create your own game modes, import custom assets, maps and completely extend it's functionalities with Lua scripts.

Before diving into modding and scripting, it's important to understand the essential concepts of how everything works in nanos world. This page will give you a brief explanation of everything you need to know and links to learn more about each topic!

After reading through this Guide, you will be ready to start programming, modding and creating content!


## Asset Packs {/* #asset-packs */}

Asset Packs are the way for modders to import custom assets into the game, such as new models, textures, sounds, and more. They can be created using Unreal Engine, exported and then added to the server's `Assets/` directory in nanos world server as an Asset Pack!

Each Asset Pack has a configuration file called `Assets.toml`. This configuration is used to list all the assets included in the pack, their paths, and other metadata such as their types, categories, and tags. This file is used by the game to load the assets and make them available for use in Lua scripts through Entities.

Through Lua Scripting, some Entities require an Asset as parameter. For example the entity <Classes.Prop /> asks for a <AssetPath.StaticMesh /> in the third constructor parameter.

This way we spawn a Prop like this, using the Asset reference:

```lua title="my-package/Server/Index.lua" showLineNumbers
-- Using the default 'nanos-world' Asset Pack, which is already natively included in the game
local my_prop = Prop(Vector(), Rotator(), "nanos-world::SM_Cube")
```

Asset Packs are a powerful tool for extending the game and adding new features, and they are essential for modders who want to create unique and exciting experiences for players. Read more about Asset Packs in our Assets Guide:

<ReferenceLink href="core-concepts/assets">Assets Guide</ReferenceLink>


## Packages {/* #packages */}

**Packages** are the core building blocks of nanos world, as they provide a way for scripters to create custom content and features for the game.

There are different types of Packages, each type with a different ability, including executing Lua **scripts**, creating **game-modes**, defining **maps**, and even creating custom **loading-screens**!

[Script](/core-concepts/packages/packages-guide.mdx#script) is the most common type of Package, it allow scripters to run Lua code on both the client and server which can listen to events, interact with game classes, and modify game behavior in various ways.

:::tip

Packages can be loaded/unloaded dynamically while the server is running with `package reload all`!

:::

Each Package is a folder under server's `Packages/` directory with a [configuration file](/core-concepts/packages/packages-guide.mdx#package-configuration) `Package.toml` inside, on which we define the type of the package and it's relevant settings.

Packages can also have dependencies on other Packages and Asset Packs, which allows scripters to reuse code and assets across multiple Packages. This can help to reduce duplication and improve organization, especially for larger scripts or game-modes.

:::tip

If we want a Package to be loaded, we must define them in our Server's [Config.toml](/core-concepts/server-manual/server-configuration.mdx), we have a section for each package type (`game_mode`, `loading_screen`, `map` and `packages`), which we must list the Package names we want the server to load!

:::

Overall, Packages are an essential concept in nanos world modding, and they provide a powerful and flexible way to create custom content and functionality for the game. Read more about them in our Packages Guide:

<ReferenceLink href="core-concepts/packages/packages-guide">Packages Guide</ReferenceLink>


## Scripting {/* #scripting */}

Scripting is a powerful way to program and customize nanos world. It allows us to listen to game events, spawn, interact, and change entities at will.

All Lua code are executed through Packages, it is possible to have scripts that run only on the server, on the client or that run on both.

### Package Folder Structure {/* #package-folder-structure */}

The folder structure of a scripting Package includes `Server/`, `Client/`, and `Shared/` folders. You can put `.lua` files in each folder to execute code on the server, client, or both (Shared):

```folder-structure
my-package/
├── Server/
│   ├── Index.lua
│   └── *.lua
├── Client/
│   ├── Index.lua
│   └── *.lua
├── Shared/
│   ├── Index.lua
│   └── *.lua
└── Package.toml
```

:::tip

Only the Packages of type `script`, `game-mode` and `map` will have this structure, as those are the only ones which run Lua scripts.

:::

The Package only loads the `Index.lua` file of each base folder (`Server/`, `Client/` or `Shared/`), this file is responsible for importing other scripting files.

You can find several guides for scripting under **`🎓 CORE CONCEPTS / Scripting`** section in the left sidebar.


### Unreal Engine Concepts {/* #unreal-engine-concepts */}

Some concepts that may sound confusing if you are not familiar with Unreal Engine. Since nanos world is built on top of it, we mostly adapt it's own concepts and terminologies.


#### Vector, Rotator and Units {/* #vector-rotator-and-units */}

In Unreal Engine, the world is represented in a 3D space using a coordinate system. The position of objects in this space is represented using <Structs.Vector />, which are essentially a set of three numbers representing the `X` (Forward), `Y` (Right), and `Z` (Up) coordinates.

The orientation of objects is represented using <Structs.Rotator />, which are a set of three numbers representing the `Pitch` (Rotation around the Y-axis), `Yaw` (Rotation around the Z-axis), and `Roll` (Rotation around the X-axis).

Unreal Engine uses **centimeters** as the default unit of measurement, so when you see a Vector with values `(100, 0, 0)`, it means that the object is 100 centimeters (or 1 meter) forward from the origin point.


#### Actors {/* #actors */}

In Unreal Engine, an **Actor** is any object that can be placed in the world. It is the base class for all objects that have a physical representation in the game world, such as <Classes.Character />, <Classes.Prop />, <Classes.Weapon />, and more. They usually can be spawned with a `location` and `rotation`.


#### Players vs Characters vs Pawn {/* #players-vs-characters-vs-pawn */}

An important concept to understand is the difference between <Classes.Player />, <Classes.Character /> and <Classes.Pawn />.

In Unreal Engine, a [**PlayerController**](https://dev.epicgames.com/documentation/unreal-engine/player-controllers-in-unreal-engine) is the entity that represents a connected player in the server, while a [**Pawn**](https://dev.epicgames.com/documentation/unreal-engine/pawn-in-unreal-engine) is the physical representation of a player in the world, which can be controlled. And [**Character**](https://dev.epicgames.com/documentation/unreal-engine/characters-in-unreal-engine) is a type of Pawn that has additional features such as walking, jumping, and more.

For nanos world, we simplified this concept a bit, and we just have a <Classes.Player /> class that represents the player. And a few character classes such as <Classes.Character /> and <Classes.CharacterSimple /> that represent the physical representation of the player in the world.

So when a <Classes.Player /> spawns, it means a real person just connected to the server.


### Logging & Debugging {/* #logging--debugging */}

You can output messages to the console using the <StaticClasses.Console /> Static Class, which provides different log levels such as <MethodReference type="StaticClass" class_name="Console" method_name="Log" />, <MethodReference type="StaticClass" class_name="Console" method_name="Debug" />, <MethodReference type="StaticClass" class_name="Console" method_name="Warn" /> and <MethodReference type="StaticClass" class_name="Console" method_name="Error" />:

```lua title="my-package/Shared/Index.lua" showLineNumbers
-- Standard console output
Console.Log("Normal scripting logs!")
Console.Debug("Debug message that will only appear if log level is >= debug!")
Console.Warn("Warning orange message! Will show the stack trace!")
Console.Error("Error red messages! Will show the stack trace!")
```


### Classes {/* #classes */}

Classes are entities that can be spawned, interacted with, and changed as you wish. An example spawning a <Classes.Light /> and <Classes.StaticMesh /> would be like this:

```lua title="my-package/Shared/Index.lua" showLineNumbers
local my_light = Light(Vector(0, 100, 100), Rotator(), Color.RED)
local my_static_mesh = StaticMesh(Vector(0, 0, 100), Rotator(), "nanos-world::SM_Cube_VR_01")
```

Or even interact and change the existing ones the way we want, such as with an existing <Classes.Prop />:

```lua title="my-package/Shared/Index.lua" showLineNumbers
local some_prop = Prop.GetByIndex(1)
some_prop:SetMesh("nanos-world::SM_Pyramid_VR")
some_prop:SetLocation(Vector(100, 100, 100))

local prop_rotation = some_prop:GetRotation()
```

Read more about Classes in the Classes Guide:

<ReferenceLink href="core-concepts/scripting/classes-guide">Classes Guide</ReferenceLink>


### Static Classes {/* #static-classes */}

Besides Classes we also have the **Static Classes**, which are just Lua libraries with functions which can be called without needing to spawn an entity or have an instance.

For example we can send a Chat message using the <StaticClasses.Chat /> Static Class:

```lua title="my-package/Server/Index.lua" showLineNumbers
Chat.BroadcastMessage("Hello world!")
```

Or doing a raycast with the <StaticClasses.Trace /> Static Class on the client side:

```lua title="my-package/Client/Index.lua" showLineNumbers
local start_location = Vector(0, 0, 100)
local end_location = Vector(1000, 0, 100)
local collision_channels = CollisionChannel.WorldStatic
local trace_mode = TraceMode.DrawDebug | TraceMode.TraceComplex

-- Do the trace
local trace_result = Trace.LineSingle(start_location, end_location, collision_channels, trace_mode)
```


### Events {/* #events */}

In nanos world, everything in the API is based on **Events**, which are similar to **Hooks** in other games.

For example, events are fired when a <Classes.Player /> spawns (<EventReference type="Class" class_name="Entity" event="Spawn" />), a <Classes.Character /> enters a vehicle (<EventReference type="Class" class_name="Character" event="EnterVehicle" />) or takes damage (<EventReference type="Class" class_name="Damageable" event="TakeDamage" />).

You can listen to these events using Lua scripting and perform actions accordingly. For example, this is how you know when a **Player** joins the server or a **Prop** is spawned:

```lua title="my-package/Shared/Index.lua" showLineNumbers
-- Subscribes when a Player joins the server
Player.Subscribe("Spawn", function(player)
	Console.Log("The player %s has joined the server!", player:GetName())
end)

-- Subscribes when a Prop is spawned
Prop.Subscribe("Spawn", function(prop)
	Console.Log("A prop id %d just spawned!", prop:GetID())
end)
```

Read more about Events in the Events Guide:

<ReferenceLink href="core-concepts/scripting/events-guide">Events Guide</ReferenceLink>


### Timers {/* #timers */}

With our <StaticClasses.Timer /> Static Class, you can create timers and delays to execute code after a certain amount of time has passed. This is useful for creating timed events, cooldowns, or any situation where you want to delay the execution of some code.

Our Timer API is similar to **JavaScript Timers**, you can use <MethodReference type="StaticClass" class_name="Timer" method_name="SetTimeout" show_class_name /> to execute a function once after a delay, or <MethodReference type="StaticClass" class_name="Timer" method_name="SetInterval" show_class_name /> to continuously execute a function repeatedly at a specified interval.

```lua title="my-package/Shared/Index.lua" showLineNumbers
-- Runs a function every 10 seconds
Timer.SetInterval(function()
    Console.Log("Triggered each 10 seconds!")
end, 10000)

-- Runs a function after 1 minute (once)
Timer.SetTimeout(function()
    -- Do Something
end, 60000)
```

Read more about Timers in the Timer Page:

<ReferenceLink href="scripting-reference/static-classes/timer">Timer Static Class</ReferenceLink>


### Finding Existing Entities {/* #finding-existing-entities */}

Besides spawning an entity and storing a variable reference to it, or getting it's reference through an Event, sometimes you need to get an already spawned entity.

For example, we can get all spawned entities from a class with their static methods <MethodReference type="Class" class_name="Entity" static_method_name="GetAll" /> or <MethodReference type="Class" class_name="Entity" static_method_name="GetPairs" />:

```lua title="my-package/Shared/Index.lua" showLineNumbers
-- Returns a copy of a table of all Characters spawned
local all_characters = Character.GetAll()
```

```lua title="my-package/Shared/Index.lua" showLineNumbers
-- Efficiently iterates over all Players spawned
for _, player in pairs(Player.GetPairs()) do
	Console.Log("Player %s is connected!", player:GetName())
end
```

Or we can get a reference to an existing <Classes.Prop /> with a specific Index with the method <MethodReference type="Class" class_name="Entity" static_method_name="GetByIndex" />:


```lua title="my-package/Shared/Index.lua" showLineNumbers
local my_prop = Prop.GetByIndex(10)
```

Another way is retrieving an entity by it's globally unique ID with the methods <MethodReference type="StaticClass" class_name="Server" method_name="GetEntityByID" show_class_name /> or <MethodReference type="StaticClass" class_name="Client" method_name="GetEntityByID" show_class_name />:

```lua title="my-package/Server/Index.lua" showLineNumbers
local my_character = Character(Vector(), Rotator(), "nanos-world::SK_Male")
local character_id = my_character:GetID()

-- Later on...
local my_character = Server.GetEntityByID(character_id)
```


### Retrieving a Player Character {/* #retrieving-a-player-character */}

As explained in [Players vs Characters vs Pawn](#players-vs-characters-vs-pawn) section, a <Classes.Player /> represents a connected player in the server, while a <Classes.Character /> is the physical representation of the player in the world.

You can retrieve the Player's possessing Character with the method <MethodReference type="Class" class_name="Player" method_name="GetControlledCharacter" show_class_name />.


#### Retrieving the Local Player on Client Side {/* #retrieving-the-local-player-on-client-side */}

On **Client** side, there are some special methods and events to retrieve the local Player: <MethodReference type="StaticClass" class_name="Client" method_name="GetLocalPlayer" show_class_name /> and <EventReference type="StaticClass" class_name="Client" event="SpawnLocalPlayer" show_class_name />:

```lua title="my-package/Client/Index.lua" showLineNumbers
local local_player = Client.GetLocalPlayer()

-- Note: the local Player is only available after the Client "SpawnLocalPlayer" event is triggered
if (local_player) then
	local local_character = local_player:GetControlledCharacter()

    -- Get my character's location
    Console.Log("My character is at location %s", local_character:GetLocation())
end
```

```lua title="my-package/Client/Index.lua" showLineNumbers
Client.Subscribe("SpawnLocalPlayer", function(player)
	Console.Log("My Player just spawned in client side! %s", player:GetName())
	-- Only now it's possible to access Client.GetLocalPlayer()
end)
```


## Networking {/* #networking */}

Our client-server architecture has been designed thinking about the ease and simplicity of everything related to networking and synchronization. Leaving the scripter to worry only with the creative part!

All entities that are spawned on the server are also automatically spawned and synchronized on the client.

When calling a function on an entity in the server side, this function is also automatically called on the same entity on every client connected. Making everyone always to be synchronized with all changes!

For example, if we spawn a <Classes.Character />, and set it's location, all Clients will have it's location updated as well:

```lua title="my-package/Server/Index.lua" showLineNumbers
-- Spawns a Character (on server side)
local my_character = Character(Vector(0, 0, 0), Rotator(), "nanos-world::SK_Male")

-- Sets it's location
my_character:SetLocation(Vector(100, 100, 100))
```

Some Classes can only be spawned on server, others only on client. The same for some functions of them. This way, if you spawn an entity in the client side, it will only exist for that client.

:::tip

Through this docs, you will find [Authority](/core-concepts/scripting/authority-concepts.mdx#authority) tooltips telling whether a function can be called on server <AuthorityType.ServerOnly />, on client <AuthorityType.ClientOnly />, on both <AuthorityType.Both /> or only on the side which spawned it <AuthorityType.AuthorityOnly />!

:::


### Communication Between Sides {/* #communication-between-sides */}

To be able to send information from the server to client or vice versa, we exposed some especial methods in our <StaticClasses.Events /> Static Class to allow that!


```lua title="my-package/Client/Index.lua" showLineNumbers
-- Subscribes to a Remote Event on client side
Events.SubscribeRemote("MyClientEvent", function(my_text)
    Console.Log("Event received from server! " .. my_text)
    -- outputs "Event received from server! hello nanos world!"
end)
```

```lua title="my-package/Server/Index.lua" showLineNumbers
-- Sends a remote Event to all Players in all Client Packages
Events.BroadcastRemote("MyClientEvent", Reliability.Reliable, "hello nanos world!")
```

Read more about Networked Events in our Communicating Between Packages Guide:

### Synchronized Values {/* #synchronized-values */}

Besides sending data through Events, which is a dynamic thing, we also have the possibility to bind/set custom values to entities, or even globally in the Server/Client.

For example we can set a value on any entity, which can be accessed by any Package:

```lua showLineNumbers
-- Sets a 'my_value' value to a Prop
my_prop:SetValue("my_value", 100)

-- Later on, get the value again
local my_value = my_prop:GetValue("my_value")
```

This will make `my_value` to be set on this entity globally in this side (client or server).

But we also have the ability to send a *synchronized* value on an entity (if we are on server side)! For that we just append the `true` parameter, telling it to be sent and synchronized to all clients too!

```lua title="my-package/Server/Index.lua" showLineNumbers
-- Sets a synchronized 'my_value' value to a Prop
my_prop:SetValue("my_value", 100, true)
```

Those values are accessible by any Package and are a great way to synchronize custom values!

## Maps {/* #maps */}

Maps (or Levels in Unreal) are the entry point of the game, this is where the world is loaded and the entities are spawned in the client side.

To configure a Map, we create a Package of type [map](/core-concepts/packages/packages-guide.mdx#map), this type is like `script`, but it's `Package.toml` has additional settings to configure the map, e.g. we have `map_asset = "my-asset-pack::MyLevel"` which we need to define which Map Asset this Package will load.

We can configure which map the server will run in the `Config.toml`, under `map = "my-map-package"` setting.

:::tip

We provide 4 built-in Maps which can be loaded without any additional Package or Asset Pack: `default-blank-map`, `default-empty-map`, `default-ocean-map` and `default-testing-map`.

:::

Also, in the map's Package.toml, we can configure `spawn_points` and `custom_data`, which can be accessed through scripting by any Package:

```lua title="my-package/Server/Index.lua" showLineNumbers
local spawn_points = Server.GetMapSpawnPoints()
local custom_data = Server.GetMapConfig()
```

As you could notice, by loading a Map you are also loading a Package with scripts, so you can have custom scripts running per-map. Which can be useful to add custom behaviors such like interactable environments or doors.


## Game Modes {/* #game-modes */}

[Game-Modes](/core-concepts/packages/packages-guide.mdx#game-mode) are a type of Package which behave exactly like `scripts`, with the exception we can just have **one** game-mode loaded at once, while we can have as many as scripts running as we want.

They should be used to create unique and self-contained game experiences.

Players can select and start servers easily by choosing a game-mode in the New Game section on Main Menu.
